## 實驗二：Multi-Call HW 驗證（Mode A）

### 背景

FA 硬體設計中，`FA_interrupt` 在計算完成後持續保持 HIGH，不會自動清除。若直接呼叫第二次，`wait_for_irq()` 會立刻返回（看到舊的 interrupt）。

### 修正前後對比

```
修正前（Bug）：
  call  1: ri.cycles=747,497  ri.read=363,776  ← 正常
  call  2: ri.cycles=8        ri.read=0        ← 立刻返回，無計算！
  call  3: ri.cycles=8        ri.read=0
  call 4-6: 同上

修正後（加入 get_fa_hal()->reset() 於每次呼叫前）：
  call  1: ri.cycles=747,497  accum=747,497
  call  2: ri.cycles=747,497  accum=1,494,994
  call  3: ri.cycles=747,497  accum=2,242,491
  call  4: ri.cycles=747,497  accum=2,989,988
  call  5: ri.cycles=747,497  accum=3,737,485
  call  6: ri.cycles=747,497  accum=4,484,982
```

### 執行指令

```bash
# 建置帶 STATS_DEBUG 的版本
make -C test/testbench/vit clean && make -C test/testbench/vit STATS_DEBUG=1
# Mode A 執行（6 heads × 1 block）
./test/testbench/vit/build/tb_vit 2>&1 | grep "FA-STATS"
```

### 結論

```
6 次 HW call 全部：
  ri.cycles  = 747,497  (每次相同，符合預期)
  ri.read    = 363,776 bytes
  ri.write   = 50,176 bytes
  accum      = N × 747,497  (精確線性累積)
```

## 實驗五：ScaleTying 比較

### 說明

ScaleTying 將 Q 和 K 使用同一個量化 scale（`max(q_scale, k_scale)`），可簡化硬體反量化路徑。

### 執行指令

```bash
make -C test/testbench/flash_attn run_scale_tying_cmp
```

### 實驗結果

```
Scale Tying comparison complete.
  [untied]  Max abs diff : 1.35e-04  at idx=12003
  [tied  ]  Max abs diff : 1.35e-04  at idx=12003
```

### 結論

```
┌──────────────────────────────────────────────────────────────┐
│  ScaleTying 對精度的影響                                       │
├─────────────────┬──────────────┬──────────────────────────── ┤
│                 │ Max abs diff │ 說明                         │
├─────────────────┼──────────────┼───────────────────────────── ┤
│ Independent Q/K │ 1.35e-04     │ 各自最優量化 scale           │
│ Scale Tying     │ 1.35e-04     │ max(q_scale, k_scale)       │
└─────────────────┴──────────────┴───────────────────────────── ┘

兩者相同的原因：
  case2 使用合成資料，Q 和 K 來自同分布 → q_scale ≈ k_scale
  → max(q_scale, k_scale) ≈ q_scale ≈ k_scale → 量化結果相同

結論：ScaleTying 在本實驗不增加誤差，且簡化硬體（1 個乘法 vs 2 個）✅
```

## 已知限制與未來工作

```
① Verilator 模擬速度
   Mode B (72 calls) 需約 1 小時
   FPGA 實際部署後將快 1000x+

② FA_interrupt 不自動清除
   目前每次 HW call 前需 hardware reset（overhead 極小但非最優）
   理想解法：在 Verilog 中，start 信號的上升沿清除 FA_interrupt

③ CLS attention 仍在 SW
   計算量：197 × 64 × 6 heads = 75,648 fp32 ops（每層），可接受
   若要純 HW：需擴展 FA 支援 N=197（197%14≠0，需 padding）

④ 準確率樣本數
   50 張樣本的統計信心區間較寬
   建議：使用 500~1000 張做最終評估
```
