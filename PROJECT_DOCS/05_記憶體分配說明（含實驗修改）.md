## 第二部分：軟體層記憶體結構（實驗過程修改）

### ★ 修改 A：CLS Token Attention 緩衝區

**修改時間**：Phase 1 整合測試期間  
**修改原因**：CLS token 的 attention 被歸零，導致準確率 0%

#### 修改前（Bug 版本）

```
vit_attention_layer() 記憶體佈局：

qkv_out[T=197, 3D=1152]    ← T×3D 的 QKV 投影輸出
concat_out[N=196, D=384]   ← Patch tokens 的注意力輸出
                             （CLS row 被忽略！attn_out[0]=0）

堆疊記憶體佔用：
  qkv_out    = 197 × 1152 × 4 = 907,264 bytes ≈ 886 KB  (heap)
  concat_out = 196 × 384 × 4  = 301,056 bytes ≈ 294 KB  (heap)
  attn_out   = 197 × 384 × 4  = 302,592 bytes ≈ 296 KB  (caller)

CLS token 問題：
  attn_out[0, :] = 0   ← 永遠為零，CLS 無法聚合資訊
  每層 12 次放大後，final logits 固定輸出相同 class（646）
```

#### 修改後（Fix 版本）

```
新增的臨時緩衝區（heap，每個 attention layer 呼叫時）：

K_all[T=197, HD=64]     ← 每個 head 暫存所有 token 的 K
                           = 197 × 64 × 4 = 50,432 bytes ≈ 49 KB
V_all[T=197, HD=64]     ← 每個 head 暫存所有 token 的 V
                           = 197 × 64 × 4 = 50,432 bytes ≈ 49 KB
o_cls_h[HD=64]          ← stack，每個 head 的 CLS attention 輸出
                           = 64 × 4 = 256 bytes（stack 上，安全）
cls_concat[D=384]       ← heap，6 heads CLS 輸出拼接
                           = 384 × 4 = 1,536 bytes
cls_attn[D=384]         ← heap，O-projection 後的 CLS 輸出
                           = 384 × 4 = 1,536 bytes

生命週期：
  K_all/V_all → 每個 head 分配 → single_query_attention() → 立刻 free
  cls_concat → H heads 填滿後 → linear() → free
  cls_attn → memcpy 到 attn_out[0] → free

每個 attention layer 額外 heap 峰值：
  ≈ 50 KB (K_all) + 50 KB (V_all) + 2 KB (其他) ≈ 102 KB
```

```
CLS attention 完整記憶體流程圖：

qkv_out[T,3D] ──┬──► concat_out[N,D] ──► patch_attn[N,D]  (O-proj)
                │                          ↓
                │                    attn_out[1:T, D]       ← Patches
                │
                └──► K_all[T,HD] ───┐
                     V_all[T,HD] ───┼──► single_query_attn() → o_cls_h[HD]
                     q_cls_h[HD] ───┘         ↓
                                         cls_concat[D]
                                              ↓
                                         cls_attn[D] (O-proj)
                                              ↓
                                    attn_out[0, D]           ← CLS ★修正
```

### ★ 修改 B：VitHWStats 結構

**修改時間**：P2 Stats 累加 Bug 修正  
**修改原因**：需要追蹤個別 call 的 cycle 數作為交叉驗證基準

#### 修改前

```c
typedef struct {
    uint64_t total_cycles;        // 累積 cycle 數
    uint64_t total_memory_read;   // 累積 DMA 讀取 bytes
    uint64_t total_memory_write;  // 累積 DMA 寫入 bytes
    uint32_t fa_call_count;       // FA 呼叫次數
} VitHWStats;

// 宣告方式：
static VitHWStats g_hw_stats;   // ← 非 volatile
```

#### 修改後

```c
typedef struct {
    uint64_t total_cycles;
    uint64_t total_memory_read;
    uint64_t total_memory_write;
    uint32_t fa_call_count;
    uint64_t last_call_cycles;    // ★ 新增：最後一次 call 的 cycle 數
                                  //   用途：total_cycles ≈ count × last_call_cycles
} VitHWStats;

// ★ 宣告改為 volatile：
static volatile VitHWStats g_hw_stats;
// 原因：-O2 優化可能把連續的 += 合並或省略，
//       volatile 強制每次讀-修改-寫都真實發生
```

**記憶體大小變化**：

```
修改前：4 × 8 + 1 × 4 = 36 bytes（對齊後 40 bytes）
修改後：5 × 8 + 1 × 4 = 44 bytes（對齊後 48 bytes）
增加：8 bytes（可忽略）
```

### ★ 修改 C：HAL 層 — Pre-Call Hardware Reset

**修改時間**：Multi-Call HW Bug 修正  
**修改原因**：`FA_interrupt` 不會自動清除，第 2 次以後的 HW call 立刻返回

#### 問題分析

```
FA_interrupt 信號生命週期（修正前）：

時間 ──────────────────────────────────────────────────────►
      init()  call1完  fa_stop  call2開始  wait_for_irq
               │        │         │               │
FA_irq:   0────┼─── 1 ──┼─── 1 ──┼─── 1 ─────── 1 ──► 永遠為 1！
                         ↑                         ↑
                    持續保持 HIGH              立刻返回 (8 cycles)

根本原因：
  fa_stop() 寫 control=0，但不清除 FA_interrupt
  fa_start() 寫 control=1，Verilog 的上升沿觸發需要幾個 cycle
  wait_for_irq() 見到 FA_interrupt=1 立刻退出
```

#### 修改方案

```
方案比較：
  ❌ 方案1：在 wait_for_irq() 加 Phase 1（等 irq→LOW）
     問題：FA_interrupt 永遠為 HIGH，造成無窮迴圈（實測卡 1 小時）

  ✅ 方案2：每次 flash_attention() 前呼叫 get_fa_hal()->reset()
     效果：ARESETn=0→1 脈衝，硬體完整重置，FA_interrupt 清為 0
     代價：每次 call 多 FA_RESET_CYCLE 個 clock steps（幾個 ns）
            但這些 cycles 在 reset_runtime_info() 前發生，不計入統計
```

#### 呼叫位置

```cpp
// src/runtime/vit/runtime_vit.cpp
static void fa_call_and_accumulate(...) {
    get_fa_hal()->reset();    // ★ 新增：清除 FA_interrupt
    flash_attention(...);     // 已有的 flash_attention 呼叫
    ...
}
```

#### 記憶體影響

```
HAL 物件記憶體大小：不變（reset() 只是函數呼叫）
額外 clock_step 次數：FA_RESET_CYCLE 次（定義在 flash_attn_hal.hpp）
對 runtime_info 的影響：reset_runtime_info() 在 flash_attention() 內
                         呼叫，清零 reset 消耗的 cycle，不影響統計
```

## 第三部分：靜態緩衝區佈局（runtime_vit.cpp）

```
runtime_vit.cpp 中的靜態全域緩衝區：

static float s_Q_h[N * HD];   N=196, HD=64, = 196×64×4 = 50,176 bytes
static float s_K_h[N * HD];   同上
static float s_V_h[N * HD];   同上
static float s_O_h[N * HD];   同上

靜態分配原因：
① 地址穩定：vm_addr_h_ 機制要求 Q/K/V/O buffer 與 HAL 物件
            在同一 4GB 虛擬地址空間內（高 32 位相同）
② 避免重複分配：196×64×4×4 = 200 KB，每個 attention layer 呼叫多次

佔用：4 × 50,176 = 200,704 bytes ≈ 196 KB（BSS segment）

static volatile VitHWStats g_hw_stats:
佔用：48 bytes（BSS segment）
```
