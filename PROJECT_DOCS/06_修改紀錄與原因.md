## 修改 #1 — Makefile：預設 target 錯誤

**修改時間**：P1 整合測試初期  
**修改檔案**：`test/testbench/vit/Makefile`

### 問題現象

```
$ make clean && make
[VIT] Cleaning...
$
# 沒有任何編譯訊息！binary 不存在
```

`make` 完成後 `./build/tb_vit` 找不到，但 `make` 沒有報錯。

### 根本原因

```makefile
# 修改前（錯誤）：
.PHONY: all hw_build run run_real run_full run_batch clean

hw_build:             ← 第一個出現的非 .PHONY target！
    @if [ ! -f "$(HW_DIR)/libVflash_attn_wrapper.a" ]; then \
        ...           ← .a 已存在 → if 條件不成立 → 直接退出
    fi

all: $(TARGET)        ← 排在第二，不是預設 target
```

GNU Make 規則：**預設 target = Makefile 中第一個非 `.PHONY` 的 target**。  
`hw_build` 排在 `all` 之前，`make` 不帶參數只執行 `hw_build`，  
而 `.a` 已存在，`hw_build` 什麼都不做直接退出。

### 修正方案

```makefile
# 修改後（正確）：
.PHONY: all hw_build run run_real run_full run_batch clean

# 'all' 必須是第一個 real target，確保裸 'make' 建置 tb_vit
all: $(TARGET)        ← 移到最前面

hw_build:             ← 移到 all 之後
    @if [ ! -f "$(HW_DIR)/libVflash_attn_wrapper.a" ]; then \
        ...
    fi

$(TARGET): $(ALL_SRCS) | hw_build
    $(CXX) $(CXXFLAGS) ... -o $@ $^
```

### 驗證

```
$ make clean && make
g++ -std=c++17 ... -o build/tb_vit ...
[VIT] Build complete: build/tb_vit
```

## 修改 #2 — CLS Token 注意力為零（Bug → Fix）

**修改時間**：P1 整合測試，0% 準確率問題  
**修改檔案**：`src/runtime/vit/runtime_vit.cpp`

### 問題現象

```
Mode D SW 批次測試結果：
  Top-1 correct: 0 / 50 (0.0%)
  所有圖片預測 class 646

Mode C 單張測試：
  Predicted class: 646   (ground truth: 0)
  *** WRONG ***
```

### 根本原因分析

```
vit_attention_layer() 原始實作（Bug 版本）：

qkv_out = linear(tokens, W_qkv)   → shape [197, 1152]  (T=197, 3D=1152)

# Patch tokens 注意力（head h）：
concat_out = HW_FA(qkv_out[1:197, h*HD:(h+1)*HD], ...)
             ↑ 只取 patch tokens（index 1..196），共 N=196 行

# CLS token：
attn_out[0, :] = 0   ← ★ 永遠為零！CLS row 被完全忽略

# 最終 attn_out：
attn_out[0, :] = 0.0 (CLS)
attn_out[1:,:]  = patch_attn (正常)
```

**傳播效應**：
```
attn_out[0] = 0 → after O-projection → still 0
              → after residual_add → token[0] ≈ original (沒更新)
              → 12 層後，CLS token 沒有聚合任何 patch 資訊
              → final_norm(token[0]) → 接近常數向量
              → linear(cls_head) → 固定輸出同一 class
```

### 修正方案

新增 `single_query_attention()`，對每個 head 單獨計算 CLS token 的注意力：

```cpp
// ★ 新增函數（在 vit_attention_layer 呼叫前聲明）
static void single_query_attention(const float* q,    // [HD]
                                   const float* K,    // [T, HD]
                                   const float* V,    // [T, HD]
                                   float* out,        // [HD]
                                   int tokens, int hd) {
    float scale = 1.0f / sqrtf((float)hd);
    // Step 1: attention scores = q · K[t] * scale
    float scores[MAX_TOKENS];
    float max_s = -FLT_MAX;
    for (int t = 0; t < tokens; t++) {
        float s = 0;
        for (int j = 0; j < hd; j++) s += q[j] * K[t*hd + j];
        scores[t] = s * scale;
        if (scores[t] > max_s) max_s = scores[t];
    }
    // Step 2: numerically stable softmax
    float sum = 0;
    for (int t = 0; t < tokens; t++) {
        scores[t] = expf(scores[t] - max_s);
        sum += scores[t];
    }
    // Step 3: weighted sum of V
    memset(out, 0, hd * sizeof(float));
    for (int t = 0; t < tokens; t++) {
        float w = scores[t] / sum;
        for (int j = 0; j < hd; j++) out[j] += w * V[t*hd + j];
    }
}
```

並在 `vit_attention_layer()` 中修改 CLS 的處理：

```cpp
// 修正後的 vit_attention_layer 關鍵段落：

// 為每個 head 收集 K_all[T,HD] 和 V_all[T,HD]，然後：
float o_cls_h[HD];
single_query_attention(q_cls_h, K_all, V_all, o_cls_h, T=197, HD=64);
// 填入 cls_concat[h*HD : (h+1)*HD]

// 最後 O-projection 後：
memcpy(attn_out[0], cls_attn, D * sizeof(float));  // ★ CLS 不再為零
```

### 記憶體佈局對比

```
修改前                          修改後
─────────────────────           ─────────────────────────────────
qkv_out[197,1152]               qkv_out[197,1152]       (不變)
concat_out[196,384]             concat_out[196,384]      (patch)
attn_out[0] = 0 ← Bug          K_all[197,64]  ← ★ 新增，每 head
                                V_all[197,64]  ← ★ 新增，每 head

                                cls_concat[384]  (heap, ★新增)
                                cls_attn[384]    (heap, ★新增)
                                attn_out[0] = cls_attn  ← ★ 修正
```

### 驗證

```
修正前：Top-1 = 0%（所有圖片輸出 class 646）
修正後：Top-1 = 72.0%（50 張測試集，PASS）
```

## 修改 #3 — VitHWStats 結構：新增 `last_call_cycles` 與 `volatile`

**修改時間**：P2 Stats 累加 Bug 調查期間  
**修改檔案**：`src/runtime/vit/runtime_vit.h`、`runtime_vit.cpp`

### 問題現象

```
FA-STATS 輸出（Bug 版本）：
[FA-STATS] call  1: ri.cycles=747497    accum=747497
[FA-STATS] call  2: ri.cycles=8         accum=747505  ← 幾乎沒加！
[FA-STATS] call  3: ri.cycles=8         accum=747513
...
[FA-STATS] call  6: ri.cycles=8         accum=747545

最終統計：
  FA calls total   : 6
  Total FA cycles  : 747545   ← 應為 6 × 747497 = 4,484,982
```

這個問題有**兩個獨立的原因**，分別修正。

### 原因 A：FA_interrupt 未清除（主因，見修改 #4）

Call 2 以後 `wait_for_irq()` 立刻返回（8 cycles），`ri.elapsed_cycle` 只有 8。  
→ 這是**硬體行為問題**，修正見修改 #4。

### 原因 B：編譯器可能省略 Stats 更新（次因）

```cpp
// 修改前（Bug 隱患）：
static VitHWStats g_hw_stats;   // 非 volatile

// 問題：-O2 下，連續的讀-修改-寫序列可能被優化
// 例如：g_hw_stats.fa_call_count += 1u 若中間沒有副作用，
//       編譯器可能延遲或合併寫入（對中斷/模擬環境特別危險）
```

### 修正方案

```c
// ★ 修改後（runtime_vit.h）：
typedef struct {
    uint64_t total_cycles;
    uint64_t total_memory_read;
    uint64_t total_memory_write;
    uint32_t fa_call_count;
    uint64_t last_call_cycles;    // ★ 新增：最後一次 call 的 cycle 數
                                  //   交叉驗證：total ≈ count × last
} VitHWStats;

// ★ 宣告為 volatile，強制每次讀-修改-寫都真實發生：
static volatile VitHWStats g_hw_stats;
```

```cpp
// ★ 修改後（runtime_vit.cpp 的 fa_call_and_accumulate）：
// 明確的 volatile 讀-修改-寫（不讓編譯器自動省略）：
uint64_t cy  = g_hw_stats.total_cycles;
uint64_t mr  = g_hw_stats.total_memory_read;
uint64_t mw  = g_hw_stats.total_memory_write;
uint32_t cnt = g_hw_stats.fa_call_count;

g_hw_stats.total_cycles       = cy  + (uint64_t)ri.elapsed_cycle;
g_hw_stats.total_memory_read  = mr  + (uint64_t)ri.memory_read;
g_hw_stats.total_memory_write = mw  + (uint64_t)ri.memory_write;
g_hw_stats.fa_call_count      = cnt + 1u;
g_hw_stats.last_call_cycles   = (uint64_t)ri.elapsed_cycle;  // ★ 新增
```

**記憶體大小變化**：

```
修改前：4×8 + 1×4 = 36 bytes（對齊後 40 bytes）
修改後：5×8 + 1×4 = 44 bytes（對齊後 48 bytes）
新增   ：8 bytes（可忽略）
```

### 交叉驗證公式

```
正確執行（修正 #4 後）：
  last_call_cycles = 747,497
  fa_call_count    = 72
  total_cycles     = 53,819,784

驗算：72 × 747,497 = 53,819,784  ✅
```

## 修改 #4 — FA_interrupt 多次呼叫 Bug（最重要的修改）

**修改時間**：P2 Multi-Call HW Bug 修正  
**修改檔案**：`src/runtime/vit/runtime_vit.cpp`  
**回退修改**：`src/hal/flash_attn_hal.cpp`（錯誤嘗試已回退）

### 問題現象

```
Mode A 測試（6 heads）：
[FA-STATS] call  1: ri.cycles=747,497  ← 正常：完整 14×14 計算
[FA-STATS] call  2: ri.cycles=8        ← Bug：幾乎立刻返回！
[FA-STATS] call  3: ri.cycles=8
...
```

### 根本原因

```
FA_interrupt 信號生命週期（修正前）：

時間 ──────────────────────────────────────────────────────────►
      init()  call1啟動  call1完成   fa_stop()  call2啟動  wait_for_irq
               │              │          │           │            │
FA_irq:   0───────── 0 ────── 1 ──── 1 ─────── 1 ────────── 1 ──► 永遠 HIGH！
                               ↑          ↑                    ↑
                          計算完成    fa_stop 只寫          立刻退出
                         IRQ = HIGH   control=0，           (8 cycles)
                                    不清除 IRQ

wait_for_irq() 邏輯：
while (!device_->FA_interrupt) {   ← call 2+ 進入時 FA_interrupt=1
    clock_step(...);                  → 條件立刻為 false → 直接退出
    handle_dma();
}
// 退出時只有約 8 個 clock 的 AXI write overhead
```

### 錯誤嘗試（Phase 1 修法，造成無窮迴圈）

```cpp
// ❌ 錯誤方案：在 wait_for_irq() 加「等 IRQ 降低」的 Phase 1
void FlashAttnHAL::wait_for_irq() {
    // Phase 1：等 FA_interrupt 降至 LOW（讓上一次的 HIGH 消失）
    while (device_->FA_interrupt) {      // ← ★ 這是無窮迴圈！
        clock_step(...);                  //   FA_interrupt 永遠不會自動降回 0
    }
    // Phase 2：等 FA_interrupt 升至 HIGH（新計算完成）
    while (!device_->FA_interrupt) {
        clock_step(...);
        handle_dma();
    }
}

// 結果：程式卡在 Phase 1，等待了 1 小時以上沒有返回
```

**失敗原因分析**：
```
Verilog RTL 中的 FA_interrupt 信號行為：
  - 計算完成時：FA_interrupt = 1（持續保持 HIGH）
  - fa_stop()（寫 control=0）：FA_interrupt 維持 HIGH，不變
  - fa_start()（寫 control=1）：有上升沿觸發，但 FA_interrupt
    清除需要新計算啟動後幾個 cycle

Phase 1 的 while (FA_irq == HIGH) 永遠無法退出，
因為沒有任何軟體操作能讓 FA_interrupt 自動降低。
只有硬體 reset 脈衝（ARESETn=0→1）才能清除 FA_interrupt。
```

### 正確修正方案

```
方案比較：

❌ 方案1（已嘗試）：在 wait_for_irq() 等 FA_irq → LOW
   結果：無窮迴圈，FA_interrupt 永遠不自動清除

✅ 方案2（最終採用）：每次 flash_attention() 前，先呼叫硬體 reset
   原理：ARESETn=0→1 脈衝 → 硬體完整重置 → FA_interrupt 清為 0
          → flash_attention 重新配置暫存器 → 新計算開始
          → wait_for_irq 正常等待新的 HIGH 信號

   reset_runtime_info() 在 flash_attention() 開頭被呼叫，
   清零了 reset 本身消耗的 cycle，不影響統計數據
```

```cpp
// ★ 修正位置：runtime_vit.cpp 的 fa_call_and_accumulate()
static void fa_call_and_accumulate(float* Q, float* K, float* V, float* O,
                                   uint32_t n, uint32_t d, uint32_t br) {
    /*
     * FA_interrupt 在計算完成後持續保持 HIGH，不會自動清除。
     * 若不 reset，call 2+ 的 wait_for_irq() 看到 FA_interrupt=1
     * 立刻退出（約 8 cycles），無法等到新計算真正完成。
     * 呼叫 reset() 發出 ARESETn 脈衝，將 FA_interrupt 清為 0，
     * 讓後續 wait_for_irq() 能正常等待新的完成信號。
     * reset 消耗的 cycles 由 flash_attention() 內的
     * reset_runtime_info() 清零，不納入統計。
     */
    get_fa_hal()->reset();   // ★ 關鍵修改：Pre-call hardware reset

    flash_attention(Q, K, V, O, n, d, br);
    // ... stats accumulation ...
}
```

### FA_interrupt 時序圖（修正後）

```
修正後的信號時序：

時間 ───────────────────────────────────────────────────────────►
      init()  call1   call1完  reset()  call2   call2完  reset()  call3
               │         │       │        │         │       │
FA_irq:   0 ──── 0 ────── 1 ───── 0 ────── 0 ──────── 1 ───── 0 ─── ...
                            ↑      ↑                    ↑      ↑
                       計算完成  ARESETn脈衝           計算完  ARESETn脈衝
                       IRQ=HIGH  清除 IRQ=0            IRQ=HIGH 清除

每次 fa_call_and_accumulate：
  reset()          → FA_interrupt = 0
  flash_attention() → 配置暫存器 → fa_start() → 計算
  wait_for_irq()   → 正常等待 FA_interrupt = 1（約 747,497 cycles）
  fa_stop()         → 寫 control=0（不清 IRQ）
  get_runtime_info() → 讀取 elapsed_cycle 等
```

## 修改 #5 — ScaleTying 實作（P3-A 新功能）

**修改時間**：P3 功能擴展，ScaleTying 比較實驗  
**修改檔案**：`src/runtime/flash_attn/runtime_flash_attn.cpp`

### 設計動機

```
標準 INT8 量化（獨立 scale）：
  q_scale = max(|Q|) / 127
  k_scale = max(|K|) / 127
  GEMM1 反量化：S_fp = S_int8 × q_scale × k_scale   ← 2 個乘法因子

ScaleTying（共用 scale）：
  qk_scale = max(q_scale, k_scale)
  q_scale  = qk_scale  （兩者相同）
  k_scale  = qk_scale
  GEMM1 反量化：S_fp = S_int8 × qk_scale²           ← 只需 1 個乘法因子

硬體優點：
  只需 1 個 scale 暫存器（FA_Q_SCALE = FA_K_SCALE），
  省去 1 個 MMIO write + 1 個乘法器
```

### 實作

```cpp
// runtime_flash_attn.cpp
#ifdef FA_SCALE_TYING
    // 取 Q 和 K 的 max scale 作為共用 scale
    float q_scale = compute_scale(Q, N * d);
    float k_scale = compute_scale(K, N * d);
    float qk_scale = fmaxf(q_scale, k_scale);
    q_scale = qk_scale;
    k_scale = qk_scale;
#else
    // 各自獨立量化（預設）
    float q_scale = compute_scale(Q, N * d);
    float k_scale = compute_scale(K, N * d);
#endif
    float v_scale = compute_scale(V, N * d);
```

### 編譯啟用方式

```bash
# Makefile 中的 SCALE_TYING 旗標：
ifeq ($(SCALE_TYING),1)
    CXXFLAGS += -DFA_SCALE_TYING
endif

# 使用：
make clean && make SCALE_TYING=1
```

### 實驗結果

```
測試：FA case2（N=196, d=64，合成資料）

           Baseline（獨立 scale）   ScaleTying（共用 scale）
─────────────────────────────────────────────────────────
Max diff         1.35e-4                  1.35e-4
結論：tied = untied（合成資料 Q≈K，兩個 scale 本來就相同）
```

**為何 tied = untied**：

```
合成測試資料：Q 與 K 使用相似分布生成
  q_scale = max(|Q|) / 127 ≈ k_scale = max(|K|) / 127

因此：
  fmaxf(q_scale, k_scale) ≈ q_scale ≈ k_scale
  ScaleTying 選取的 qk_scale 與原本的兩個 scale 幾乎相同
  量化誤差沒有增加
```

**結論**：ScaleTying 對合成資料無損失，實際資料若 Q/K 分布差異大則可能略降精度。

## 修改 #6 — RMSNorm 實作（P3-B 新功能）

**修改時間**：P3 功能擴展  
**修改檔案**：`src/runtime/vit/vit_ops.cpp`、`runtime_vit.cpp`

### 設計動機

```
LayerNorm（原始）：
  mean = mean(x)
  std  = sqrt(var(x) + eps)
  y = (x - mean) / std × gamma + beta
  需要：mean subtraction、beta 參數

RMSNorm（新增）：
  rms = sqrt(mean(x²) + eps)
  y = x / rms × gamma
  不需要：mean subtraction、beta 參數
  優點：計算量較少，部分 Transformer 模型（如 LLaMA）使用
```

### vit_ops.cpp — rms_norm 實作

```cpp
// ★ 新增函數
void rms_norm(float* x, const float* gamma, int N, int d, float eps) {
    for (int i = 0; i < N; i++) {
        float* row = x + i * d;
        // Step 1：計算 RMS = sqrt(mean(x²) + eps)
        float sumsq = 0.0f;
        for (int j = 0; j < d; j++) sumsq += row[j] * row[j];
        float inv_rms = 1.0f / sqrtf(sumsq / (float)d + eps);
        // Step 2：正規化 × gamma（無 mean subtraction、無 beta）
        for (int j = 0; j < d; j++) row[j] = row[j] * inv_rms * gamma[j];
    }
}
```

### runtime_vit.cpp — NORM Macro

```cpp
// ★ 新增 Macro（runtime_vit.cpp 頂部）
#ifdef VIT_USE_RMSNORM
#define NORM(x, gamma, beta, N, d, eps) \
    rms_norm((x), (gamma), (N), (d), (eps))
#else
#define NORM(x, gamma, beta, N, d, eps) \
    layer_norm((x), (gamma), (beta), (N), (d), (eps))
#endif

// 使用（transformer_block 內）：
NORM(x_norm, w->norm1_gamma, w->norm1_beta, T, D, 1e-6f);
```

### 注意：RMSNorm 需要對應權重

```
標準 ViT-Small/16 (timm) 使用 LayerNorm，有 gamma 和 beta。
RMSNorm 版本忽略 beta，使用相同的 gamma 參數。

實測結果（50 張測試集）：
  LayerNorm：Top-1 72.0%，Top-5 92.0%
  RMSNorm  ：Top-1 72.0%，Top-5 92.0%   ← 相同！

原因：LayerNorm 的 beta（偏移量）對 ViT-S 貢獻小，
      且測試集只有 50 張，統計誤差可能掩蓋細微差異。
      若使用 RMSNorm 專屬訓練的模型，差異可能更顯著。
```

def detect_numbered_dirs(root: Path) -> bool:
    """偵測是否為編號資料夾格式（≥10 個全數字子目錄）"""
    digit_dirs = [e for e in root.iterdir()
                  if e.is_dir() and e.name.isdigit()]
    return len(digit_dirs) >= 10
