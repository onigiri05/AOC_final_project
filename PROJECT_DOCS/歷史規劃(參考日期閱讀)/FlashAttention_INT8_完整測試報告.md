### 2-2 硬體加速器內部 Dataflow（每個步驟詳細說明）

```
 DRAM（存放 fp32 格式的 Q, K, V）
   │
   │  AXI4 Master DMA — 每次讀一列（ARLEN = d-1 = 63，64個fp32字）
   │
   ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 1：DMA 讀 Q-tile（每個 i-tile 只讀一次）                           │
│                                                                         │
│  Q_buf[Br×d] = fp64 tile buffer                                         │
│  │                                                                      │
│  │  Q[i_row .. i_row+Br-1][:]  (Br=14 列 × d=64 元素)                  │
│  │  每列 = 1 次 AXI burst (ARLEN=63, 64 個 fp32 字)                     │
│  │  共 Br=14 次 burst                                                   │
│  │                                                                      │
│  │  dpi_fp32_bits_to_real() 轉換：fp32位元型態 → fp64 浮點               │
│  │  原因：行為模擬用 fp64 保持精度，防止 softmax 累積誤差                  │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │ Q_buf 完整載入後進入 j-tile 迴圈
                      │
                      ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  STEP 2：初始化 O/l/m（每個 i-tile 重置一次）                             │
│                                                                         │
│  O_buf[Br×d] ← 0.0   （本 i-tile 的輸出累加器）                          │
│  l_buf[Br]   ← 0.0   （softmax 分母，即 Σ exp(s_ij) 的累積）             │
│  m_buf[Br]   ← -∞    （行最大值，用於數值穩定的 online softmax）           │
│                                                                         │
│  為什麼需要 m 和 l？                                                      │
│  標準 softmax 需要先看完整行才能計算分母。                                  │
│  Online softmax 每看完一個 j-tile 就更新，等效但記憶體需求少。              │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │ 開始對所有 j-tile 迴圈
                      │
               ┌──────▼──────┐
               │  j-tile 迴圈 │  共 N/Br = 196/14 = 14 次
               └──────┬──────┘
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 3：DMA 讀 K-tile 和 V-tile（每個 j-tile 重讀一次）                  │
│                                                                         │
│  K_buf[Br×d]：K[j_row .. j_row+Br-1][:] — 14次 burst                   │
│  V_buf[Br×d]：V[j_row .. j_row+Br-1][:] — 14次 burst                   │
│                                                                         │
│  ⚠ 重要：K 和 V 每個 i-tile 都要重讀！                                   │
│  Case2 中，K/V 各被讀 14 次（N/Br = 14 個 i-tile）                       │
│  K 實際讀取量：702,464 B（理想 50,176 B → 放大 14×）                      │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 4：INT8 量化（FA_SA_G1_LD，1 clock cycle）                          │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  計算三個 tile 的量化尺度（在 fp64 空間中掃描最大絕對值）           │   │
│  │                                                                  │   │
│  │  q_scale = max_abs(Q_buf[0..Br-1][0..d-1]) / 127.0             │   │
│  │  k_scale = max_abs(K_buf[0..Br-1][0..d-1]) / 127.0             │   │
│  │  v_scale = max_abs(V_buf[0..Br-1][0..d-1]) / 127.0             │   │
│  │                                                                  │   │
│  │  量化公式（對稱量化，四捨五入後截斷到 [-127, 127]）:               │   │
│  │  Q_int8[i][k] = clamp( round(Q_buf[i][k] / q_scale), -127,127) │   │
│  │  K_int8[j][k] = clamp( round(K_buf[j][k] / k_scale), -127,127) │   │
│  │                                                                  │   │
│  │  例：Q_buf 最大值 ≈ 0.3，q_scale = 0.3/127 ≈ 0.00236           │   │
│  │       Q_buf[2][1] = 0.2998 → Q_int8 = round(0.2998/0.00236)   │   │
│  │                             = round(127.03) = 127              │   │
│  └──────────────────────────────────────────────────────────────┘   │
│  結果：sa_a_reg[Br×d]（INT8）和 sa_b_reg[Br×d]（INT8）                 │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 5：GEMM1 — SA 計算 Q × K^T（INT8 × INT8 → INT32）                 │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │           14×14 Systolic Array（INT8 精度版本）                    │   │
│  │                                                                  │   │
│  │   Q_int8 [14×64]   K_int8^T [64×14]                             │   │
│  │   ─────────────  ×  ─────────────   =  S_int32 [14×14]          │   │
│  │    a_mat[i][k]       b_mat[j][k]         out[i][j]              │   │
│  │                                                                  │   │
│  │   SA 時序模型（staggered pipeline）：                              │   │
│  │   Latency = 2×(14-1) + depth = 26 + d cycles                    │   │
│  │                                                                  │   │
│  │   Case2 (d=64): 2×13 + 64 = 90 cycles                           │   │
│  │                                                                  │   │
│  │   INT8 乘加：acc += int(a[i][k]) × int(b[j][k])                  │   │
│  │   最大值：64 × 127 × 127 = 1,032,256 ← 不會 INT32 overflow        │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  等待 SA done → FA_SA_G1_WT 狀態，接收 sa_out[14×14]（INT32）            │
│                                                                         │
│  反量化（dequantize）回 fp64：                                            │
│  S_buf[i][j] = real(sa_out[i][j]) × q_scale × k_scale / sqrt(d)        │
│              = INT32值 × 0.00236 × 0.00236 / 8.0                        │
│                                                                         │
│  為何要 / sqrt(d)？                                                       │
│  注意力分數 S = QK^T / sqrt(d)，防止 d 大時內積值過大造成 softmax 飽和    │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │ S_buf[Br×Br] = fp64 注意力分數矩陣
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 6：Online Softmax（FA_SA_SFX，每 clock 處理 1 列，共 Br=14 cycles） │
│                                                                         │
│  對 S_buf 的每一列（每個 query token 的注意力分數行）執行：                  │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  第 r 列（每 1 clock）：                                          │   │
│  │                                                                  │   │
│  │  ① 找本列最大值：row_max = max(S_buf[r][0..Br-1])               │   │
│  │                                                                  │   │
│  │  ② 更新 online softmax 元資料：                                   │   │
│  │     m_new = max(m_buf[r], row_max)                               │   │
│  │     corr  = exp(m_buf[r] - m_new)   ← 跨 j-tile 校正因子        │   │
│  │                                                                  │   │
│  │  ③ 計算本 tile 的注意力機率（blocking = 立即可見）：               │   │
│  │     P_buf[r][c] = exp(S_buf[r][c] - m_new)  for c=0..Br-1      │   │
│  │     sum_p = Σ P_buf[r][c]                                        │   │
│  │                                                                  │   │
│  │  ④ 用校正因子縮放 O_buf（將舊 j-tile 的貢獻折扣）：               │   │
│  │     O_buf[r][:] = O_buf[r][:] × corr                            │   │
│  │                                                                  │   │
│  │  ⑤ 更新 l, m：                                                   │   │
│  │     l_buf[r] = l_buf[r] × corr + sum_p                          │   │
│  │     m_buf[r] = m_new                                             │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  最後一列（r=Br-1）完成後，計算：                                          │
│  p_scale = max_abs(P_buf[0..Br-1][0..Br-1]) / 127.0                    │
│           （為 GEMM2 量化 P 準備）                                        │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │ P_buf[Br×Br] = fp64 注意力機率矩陣
                      │ p_scale 已計算完畢
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 7：GEMM2 — SA 計算 P × V（分 ceil(d/Br) = 5 個 chunk）             │
│                                                                         │
│  V 的 d=64 欄切成 ceil(64/14)=5 個 chunk                                │
│  Chunk 0-3: 14 欄，Chunk 4: 8 欄                                        │
│                                                                         │
│  每個 chunk（FA_SA_G2_LD → G2_ST → G2_WT → G2_NX）：                    │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  G2_LD：量化 P 和 V_chunk → INT8（1 cycle）                      │   │
│  │    P_int8[i][k] = clamp(round(P_buf[i][k] / p_scale), -127,127) │   │
│  │    V_int8[j][k] = clamp(round(V_buf[k][col+j] / v_scale),-127,127)│  │
│  │    (轉置：sa_b_reg[j][k] = V_buf[k][col_base+j])                │   │
│  │                                                                  │   │
│  │  G2_ST：觸發 SA 計算（1 cycle）                                   │   │
│  │                                                                  │   │
│  │  G2_WT：等待 SA 完成（2×13+14 = 40 cycles，depth=Br=14）         │   │
│  │    SA 計算：ΔO_int32[i][j] = Σ_k P_int8[i][k] × V_int8[j][k]   │   │
│  │    最大值：14 × 127 × 127 = 225,806 ← 不會 INT32 overflow         │   │
│  │                                                                  │   │
│  │  G2_NX：反量化 + 累加到 O_buf（1 cycle）                          │   │
│  │    O_buf[i][col+j] += real(ΔO_int32[i][j]) × p_scale × v_scale  │   │
│  └──────────────────────────────────────────────────────────────┘   │
│  5 個 chunk 完成後，進入 NEXT_J                                          │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │ O_buf 已累加本 j-tile 的貢獻
                      │
               ┌──────▼──────┐
               │ 還有 j-tile? │──YES──┐
               └──────┬──────┘       │
                      │ NO           │（回到 STEP 3）
                      │◄─────────────┘
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 8：正規化（FA_FINALIZE，1 cycle）                                   │
│                                                                         │
│  O_buf[r][k] = O_buf[r][k] / l_buf[r]   for r=0..Br-1, k=0..d-1       │
│                                                                         │
│  為什麼要除以 l？                                                          │
│  Online softmax 在計算 P 時用的是未正規化的 exp()。                        │
│  l_buf 累積了所有 j-tile 的 exp() 和（即 softmax 的分母）。               │
│  最後一次除法才是真正的 softmax 正規化。                                    │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────────────────────┐
│  STEP 9：DMA 寫回 O（FA_DMA_O_AW/W/B）                                  │
│                                                                         │
│  fp64 O_buf → fp32 格式 → DMA 寫回 DRAM                                 │
│  dpi_real_to_fp32_bits()：fp64 → fp32 位元型態                           │
│  每列 1 次 AXI Write burst（AWLEN=63，64個fp32字）                        │
│  共 Br=14 次 burst                                                       │
│                                                                         │
│  寫入量（Case2）：N×d×4 = 196×64×4 = 50,176 bytes（無放大）               │
└─────────────────────┬───────────────────────────────────────────────────┘
                      │
               ┌──────▼──────┐
               │ 還有 i-tile? │──YES──┐
               └──────┬──────┘       │
                      │ NO           │（回到 STEP 1）
                      │◄─────────────┘
                      ▼
              FA_DONE：觸發中斷
              O[196×64] 完整寫回 DRAM
```

### 3-2 14×14 Systolic Array 內部原理

```
  輸入：a_mat[i][k]（INT8）和 b_mat[j][k]（INT8）
  輸出：out_mat[i][j]（INT32）= Σ_k a[i][k] × b[j][k]

  Staggered（錯位）輸入排程（depth=4 為例）：

  時間軸 ──────────────────────────────────────►
  cycle:    0    1    2    3    4    5    6    7

  PE(0,0) 在 cycle 0 拿到 a[0][0]×b[0][0]
  PE(0,1) 在 cycle 1 拿到 a[0][0]×b[1][0]（延遲 1）
  PE(1,0) 在 cycle 1 拿到 a[1][0]×b[0][0]（延遲 1）
  PE(1,1) 在 cycle 2 拿到 a[1][0]×b[1][0]（延遲 2）

  錯位讓每個 PE 不需要等待前一個 PE，形成流水線。
  最後一個 PE(13,13) 在 cycle 2×13=26 後才開始接收資料。

  ◄───── 26 cycles stagger ─────►◄── depth cycles ──►
  總 latency = 2×(14-1) + depth

  GEMM1（depth=64）: 2×13+64 = 90 cycles
  GEMM2（depth=14）: 2×13+14 = 40 cycles

  INT8×INT8→INT32 累加：
  acc[i][j] = Σ_{k=0}^{depth-1} int(a[i][k]) × int(b[j][k])
  最大值（GEMM1）: 64 × 127 × 127 = 1,032,256 < INT32_MAX (2,147,483,647) ✓
```

## 4. INT8 量化原理

### 4-1 為什麼要量化？

```
┌─────────────────────────────────────────────────────────────┐
│             fp32（原版）vs INT8（量化版）                     │
│                                                             │
│  fp32：32 bits / 元素                                        │
│  ┌──┬────────────────────────────────────────────────────┐  │
│  │S │      exponent (8b)      │      mantissa (23b)      │  │
│  └──┴────────────────────────────────────────────────────┘  │
│                                                             │
│  INT8：8 bits / 元素                                         │
│  ┌──┬─────────────────────────────────────────────────────┐ │
│  │S │         value (-128 ~ +127)                         │ │
│  └──┴─────────────────────────────────────────────────────┘ │
│                                                             │
│  好處：                                                      │
│  • 資料量縮小 4× → DMA bandwidth 需求降低 4×（實際硬體）      │
│  • 乘法器面積縮小 ~4× → 更省功耗                             │
│  • INT8 乘法器比 fp32 快且耗電量低                            │
│                                                             │
│  代價：精度損失（量化誤差）                                    │
└─────────────────────────────────────────────────────────────┘
```

### 4-2 對稱量化流程（Symmetric Quantization）

```
以 Q-tile 為例（Case2: Q 值範圍約 [-0.3, +0.3]）

Step 1: 找最大絕對值
        max_abs = max(|Q_buf[0..13][0..63]|) ≈ 0.3

Step 2: 計算 scale
        q_scale = max_abs / 127 = 0.3 / 127 ≈ 0.00236

Step 3: 量化（round to nearest + clamp）
        Q_int8[i][k] = clamp(round(Q_buf[i][k] / 0.00236), -127, 127)

        Q_buf = 0.2998 → 0.2998/0.00236 = 127.03 → round → 127 ← 最大值
        Q_buf = 0.0    → 0.0/0.00236    = 0.0    → round → 0
        Q_buf = -0.15  → -0.15/0.00236  = -63.6  → round → -64

Step 4: 量化誤差（每個元素）
        誤差 ≤ 0.5 × q_scale = 0.5 × 0.00236 ≈ 0.00118

Step 5: SA 計算後反量化
        S_dequant[i][j] = real(S_int32[i][j]) × q_scale × k_scale / √d

    數值流動圖：
    ─────────────────────────────────────────────────────────
    Q_buf(fp64 ≈0.3) ──÷q_scale──▶ ~127 ──round──▶ Q_int8(127)
                                                        │ INT8×INT8
    K_buf(fp64 ≈0.3) ──÷k_scale──▶ ~127 ──round──▶ K_int8(127)─┘
                                                        │
                                                   S_int32 ≈ 64×127²
                                                        │ ×q_scale×k_scale/√d
                                                   S_fp64 ≈ 真實值 ±誤差
    ─────────────────────────────────────────────────────────
```

### 6-2 K/V 讀取放大的本質原因

```
FlashAttention-2 演算法的記憶體存取模式：

  外層迴圈：for i in range(N/Br):          ← 14 次
    讀 Q[i-tile]
    for j in range(N/Br):                  ← 14 次（每個 i-tile 都跑一遍）
      讀 K[j-tile]  ◄── 每次 i-tile 都要重讀所有 K
      讀 V[j-tile]  ◄── 每次 i-tile 都要重讀所有 V
      compute...
    寫 O[i-tile]

  K 每行被讀的次數 = N/Br = 14（等於 i-tile 數量）

  傳統標準 Attention（非 Flash）：
  S = Q × K^T / sqrt(d)   ← 一次算完，K 只讀一次
  P = softmax(S)           ← 但 S[N×N] 需要 N²×4 bytes = 196²×4 ≈ 154 KB
  O = P × V                ← 一次算完，V 只讀一次

  FlashAttention 的取捨：
  犧牲 K/V 重複讀（多讀 14 倍） → 換取不需要存 S[196×196] 在 SRAM
  Case2: S 矩陣大小 = 196²×4 = 153,664 bytes ≈ 150 KB
  但我們只需要 S_buf[14×14] = 784 bytes（小 196 倍）
```

### 7-2 為何 Case2 誤差反而最小？

```

               P_int8 只有極少數非零元素
               量化 P=1.0 → P_int8=127 精確，但其他 P≈0 的量化誤差相對大
               O 誤差：5.09e-04（最大）

               p_scale = max(P)/127 很小，但所有 P 值都在合理量化範圍內

               O 誤差：1.35e-04（最小）

  直觀解釋：
  N=4 時 softmax 像 [1.0, 0.0, 0.0, 0.0]，量化後 [127, 0, 0, 0]
  N=196 時 softmax 像均勻分布，量化誤差更均勻

  │  SA 利用率（活躍期）: 75.6 / 196 = 38.6%              │
  │                                                         │
  │  整體 SA 利用率: 1.81 / 196 = 0.92%                   │
  │  （低，因為 DMA 佔 97.61% 的時間，SA 閒置等資料）       │
  └─────────────────────────────────────────────────────────┘

  為何 SA 活躍期利用率只有 38.6%？
  ──────────────────────────────────────────────────────────
  GEMM1（SA_SIZE=14, depth=64）：
    90 cycles 中，前 26 cycles 是 stagger（空泡），
    有效計算 64 cycles，14×14 PEs 全開
    有效 MAC 密度 = 64/90 = 71%

  GEMM2 chunk（SA_SIZE=14, depth=14，active_cols 平均 12.8）：
    40 cycles 中，前 26 cycles stagger
    有效 MACs = 14 × 12.8 × 14 = 2,508.8
    期間 MACs 密度 = 2,508.8/(40×196) = 31.9%

  加權平均：(71% + 5×31.9%) / 6 ≈ 38.5% ≈ 量測的 38.6% ✓

  提升 SA 利用率的方法：
  • 更大的 N/d：讓 depth 遠大於 stagger（26 cycles）
  • DMA pipeline（K/V Ping-Pong），讓 DMA 不再是等待瓶頸
```

### 8-3 硬體的真正優勢在哪裡？

```
  本次模擬（fp32 DMA）無法直接量化硬體 vs CPU 的速度比。
  原因：模擬是行為模擬，CPU 時序不在 log 中。

  但可以明確的是：

  ① INT8 計算密度優勢（真實硬體實現後）：
     • fp32 Systolic Array: 196 fp32 MAC/cycle
     • INT8 Systolic Array: 相同面積可實現 ~4× 更多 MAC（INT8 乘法器更小）
     • 潛力: 196×4 = 784 INT8 MAC/cycle

  ② 記憶體頻寬優勢（實際 INT8 DMA，本次未實作）：
     • fp32 DMA: 每元素 4 bytes
     • INT8 DMA: 每元素 1 byte → 4× 頻寬節省
     • Case2 K/V 讀取: 1,404,928 bytes → 351,232 bytes（節省 1,053,696 bytes）

  ③ FlashAttention SRAM 節省（vs 標準 Attention）：
     • 標準 Attention 需要 S[196×196] = 150 KB SRAM
     • 本設計只需 S_buf[14×14] + P_buf[14×14] = ~3.1 KB
     • 節省 ~98.0% SRAM 使用量

  ④ 目前瓶頸（DMA 97.61%）：
     若實現 K/V Ping-Pong Pipeline（每個 j-tile 計算時預取下一個 j-tile），
     DMA 與計算重疊，理論最大加速比 ≈ 1/（1-0.9761×overlap_ratio）
     若 overlap 80% → 加速比 ≈ 4.2×
```

## 10. 結論與改進方向

### 10-1 本次實作完成的內容

```
  ✓ 14×14 INT8 Systolic Array（systolic_array.sv）
    • byte signed 輸入 × byte signed 輸入 → int 輸出
    • 無 DPI-C，純 SV 整數運算
    • 同樣的 cycle-accurate timing model（2×13+depth）

  ✓ INT8 量化路徑（flash_attn_wrapper.sv）
    • 對稱量化 + per-tile scale（q/k/v/p 四個 scale 暫存器）
    • fp_to_int8() function（round-half-away-from-zero + clamp）
    • GEMM1 後 INT32→fp64 反量化
    • SFX 最後一列計算 p_scale
    • GEMM2 前量化 P 和 V
    • GEMM2 後 INT32→fp64 反量化

  ✓ 三個 Case 全部驗證通過（誤差遠低於 5% 閾值）
```

### 10-2 目前的主要瓶頸

```
  瓶頸 1：DMA 佔 97.61% 時間（Case2）
  ──────────────────────────────────────────
  根本原因：K/V 被重複讀 14× 倍，且 DMA 和 SA 計算是序列的
  改進方案：K/V Ping-Pong Pipeline
    • 計算 tile pair(j) 的同時，DMA 預取 tile pair(j+1) 的 K/V
    • 需要兩組 K/V buffer（已在 final project 規格中設計）
    • 理論：若 DMA 時間 > SA 計算時間（目前是 40× 倍），
            Pipeline 可以讓 DMA 完全隱藏在 SA 計算後面

  瓶頸 2：SA 活躍期利用率 38.6%（非 100%）
  ──────────────────────────────────────────
  根本原因：Stagger 延遲（26 cycles）佔 GEMM2 latency（40 cycles）的 65%
  改進方案：GEMM2 用更大的 depth（增大 Bc）
    • 若 Bc=64（目前 Bc=Br=14），GEMM2 depth=64，stagger 比例下降
    • 需要更大的 V buffer

  瓶頸 3：DMA 格式仍為 fp32（本次未實作 INT8 DMA 打包）
  ──────────────────────────────────────────
  改進方案：4個 INT8 打包成 1個 32-bit AXI word
    • ARLEN_M = d/4-1 = 15（目前 63）
    • DMA 讀取量縮小 4× → 讀頻寬提升 4×
    • 實際硬體中這是 INT8 最大的收益
```

### 10-3 如果做了改進，預期效果（基於實測數據估算）

```
  基線（目前）：  Case2  2,717,381 cycles

  改進A：INT8 DMA 打包（4 INT8/word）
          DMA 讀取量 1,455,104 → 363,776 bytes（縮小 4×）
          若 DMA bandwidth 不變，DMA 時間 ×0.25
          新 DMA cycles ≈ 2,652,309 × 0.25 = 663,077
          總 cycles ≈ 663,077 + 65,072 = 728,149
          加速比 ≈ 2,717,381 / 728,149 ≈ 3.7×

  改進B：K/V Ping-Pong（DMA 與計算重疊）
          SA compute: 65,072 cycles（不變）
          DMA: 已被 SA 計算隱藏（若 DMA << SA，完全隱藏）
          但目前 SA << DMA（DMA=2,652,309 >> SA=65,072）
          Ping-Pong 需先做改進A後才有意義

  改進A+B（組合）：
          每個 tile pair 的關鍵路徑 = max(DMA_time, SA_time)
          INT8 DMA time per tile pair ≈ 663,077/196 ≈ 3,383 cycles
          SA time per tile pair ≈ 332 cycles
          Ping-Pong 後：關鍵路徑 = 3,383 cycles（DMA 仍主導）
          總 cycles ≈ 196 × 3,383 + startup ≈ 662,000 cycles
          vs 基線 2,717,381 → 加速比 ≈ 4.1×

  注：以上估算基於實測數據，假設 DMA bandwidth 線性縮放
```
